## 1. Executive summary

The codebase is healthy at the baseline: ruff clean, mypy strict clean
across `src/` and `tests/` (123 source files), 565 pytest tests green
before this engagement. The merge train since 2026-05-27b closed the
two live-deployment findings that had carried for a week (the
stale-build freeze and the per-request engine reboot), so the audit
backlog is now dominated by the two unimplemented ADRs (0021, 0022)
rather than by live-VM state.

This engagement lands one backlog item end to end. BL-016 (audit JSONL
hash chain) was the oldest unstarted L1 item and the one most aligned
with the simulator's first value, auditability. It ships as an additive
change to `audit.py` behind ADR 0025: each line now commits to its
predecessor, a `verify_chain` walk locates the first tampered line, and
a T0 `audit_verify` tool exposes the check to a controller. The audit
trail is now self-verifying for mutation and mid-stream deletion.

The audit also found documentation drift that the prior cadence missed:
`LIMITATIONS.md` still declares "No hot-reload of profiles" although the
`profile_reload` tool (BL-039) shipped, an L-row cites the wrong BL id,
and both `STATUS.md` and `LIMITATIONS.md` carry stale tool and test
counts (nineteen tools and 351 tests against an actual twenty-nine and
578). These are corrected in this PR and recorded below.

## 4. New observations from this engagement

### N19 (Low). The SQLite `audit_entries` mirror is never written

`db.py` defines `AuditEntry` and the Alembic baseline
(`alembic/versions/0001_baseline.py`) creates the `audit_entries`
table, per ADR 0002 ("SQLite with Alembic, JSONL audit alongside").
Nothing instantiates `AuditEntry`: the live audit trail is the JSONL
sink only, and the SQLite mirror is dead schema. This is not a defect
(the JSONL is authoritative and is the off-host artefact), but the
advertised "alongside" mirror does not exist in practice, which a
reader of ADR 0002 or the schema would not expect. Tracked as BL-065:
either wire the mirror (and carry the BL-016 hash fields into it) or
record in ADR 0002 that the SQLite audit table is reserved, not live.

### N20 (Informational). The hash chain is JSONL-only

The BL-016 chain (this engagement) covers the JSONL sink. If N19 is
resolved by wiring the SQLite mirror, the mirror rows should carry
`prev_hash` / `entry_hash` so both surfaces are verifiable. Recorded
in ADR 0025's consequences and folded into the BL-065 description.

## 6. Recommended remediation order

What remains, in priority:

1. **ADR 0021 implementation.** Per-subsystem tool modules under
   `src/nous/tools/`. `server.py` has grown to 1079 lines; the split
   is now the clearest structural win. Pure refactor, scoped per
   module group, with the tier-classifier coverage test the ADR calls
   for.
2. **ADR 0022 foundation.** Land `src/nous/safety/enforcer.py`
   (`SafetyResult` plus `SafetyEnforcer`) as the additive increment,
   then wire the FSM thermal guard (SC-2) through it as the first
   caller. The new `Tier.SAFETY` and the `audit.py` field touch a
   boundary, so the wiring PR carries its own security note.
3. **BL-065 (N19).** Decide the SQLite audit mirror: wire it with the
   hash fields, or document it as reserved in ADR 0002.
4. **BL-031 daily anchor.** Record (or sign) the chain head per UTC
   day to close the tail-truncation gap BL-016 leaves open.

## 7. Out of scope

Items the prior baselines excluded remain out of scope: mesh / DTN
absence, parametric biometrics by design, absence of a real local
model, the FSM raising on an unknown trigger, and `audit.write`
swallowing its own exceptions (the best-effort posture is deliberate;
the new hash computation is wrapped in the same posture).
