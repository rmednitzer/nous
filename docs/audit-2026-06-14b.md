# Audit 2026-06-14b -- fresh adversarial pass post BL-048 / BL-088

This is a full fresh audit at HEAD (`104011d`, after the BL-048 / ADR 0053
propagation-aware comms model, the BL-088 / ADR 0054 higher-fidelity link
budget, and the BL-087 / ASSESS-1 self-model fix). The 2026-06-14 audit
(`audit-2026-06-14.md`) covered the pre-BL-048 codebase and is fully closed; the

adversarial code review of the new comms and propagation surfaces, and an
adversarial review of the core and high-blast surfaces, plus a documentation and
comment drift sweep.

## Findings: comms and propagation (the new BL-048 / BL-088 code)

### H-1 (HIGH, open, BL-091) -- `tx` accepts bytes on a zero-capacity link

`subsystems/comms.py` `tx()` caps the achieved rate at `capacity_bps` (BL-048),
but when a propagation link is driven below `snr_floor_db` its `capacity_bps` is
`0.0`, and `tx` still returns `n_bytes` "accepted" while stamping
`throughput_bps = 0`. The accepted-byte count is then falsely optimistic: a
direct `comms_send` call on a fully-degraded link reports success. `flush_tick`
is unaffected (a zero capacity budget defers the package), but the direct `tx`
path is not. The fix is to gate `capacity_bps <= 0.0` to `return 0`, consistent
with the existing `forced_state is False` guard. Low-blast (`subsystems/comms.py`
is not a high-blast surface); a regression test pins it.

### H-2 (LOW after validation, open, BL-091) -- estimator `setdefault` freezes a missing link's estimate

`estimators/comms.py` `update()` refreshes a link absent from the current
observation with `self._link_estimates.setdefault(missing, ...)`, which only
writes when the key is absent, so after the first tick a genuinely-missing link's
estimate is frozen while its particle belief drifts. Rated HIGH by the sweep, but
validated LATENT: `CommsSubsystem.sensor_obs()` always emits every link it holds,
and a profile reload rebuilds both the subsystem and the estimator, so the
`missing` set is always empty through the engine. The fix (an unconditional
assignment) is a one-line correctness improvement with no live impact; folded
into BL-091.

### H-3 (reviewed, working as designed) -- outbox loss closes the link for the flush

`state/comms_outbox.py` `_delivery_lost` draws one Bernoulli(loss) sample for the
head package on a link and, on a loss, closes the link for the rest of the flush.
The sweep flagged this as under-delivering (one draw per link per tick rather
than per package). Reviewed as intentional: closing on the first loss is exactly
what preserves the precedence invariant, because the packages are processed in
descending precedence, so anything after a deferred package is equal or lower and
must not jump ahead. The consequence is a head-of-line drain on a lossy link
(the head clears at rate `1 - loss` per tick), which is the conservative,
precedence-correct behaviour. A future refinement could keep drawing within a
single precedence band; noted, not scheduled.

### M-1 (LOW, open, BL-091) -- estimator `LinkEstimate` omits `bandwidth_bps`

`_link_estimate_from_belief` never sets `bandwidth_bps`, so an estimator-produced
`LinkEstimate` carries `bandwidth_bps = 0` and `comms_state.derive` falls back to
the legacy flat throughput floor for it rather than the per-link capacity
fraction. The FSM-facing path is unaffected (`derive_state` consumes the
subsystem's estimates, which do carry `bandwidth_bps`); the estimator's estimates
are informational. Folded into BL-091 for consistency.

### M-2 (LOW, open, BL-091) -- `_likelihood_given_connected` uses `<=` against the throughput floor

A link observed at exactly `_THROUGHPUT_FLOOR_BPS` (1 bps) early-returns the
likelihood floor (treated as disconnected). A 1 bps link is dead for every
practical purpose, so the `<=` is defensible, but `<` would be cleaner and match
the "below the floor" intent. Cosmetic; folded into BL-091.

### M-3 (MED, open, BL-095) -- `comms_state` CONNECTED requires every configured link healthy

`derive` returns CONNECTED only when `len(healthy) == len(links)`, where `healthy`
is drawn from connected links and `links` is the full inventory. A multi-link
profile with a backup link that has aged out can therefore never report
CONNECTED, only LIMITED. This is pre-existing (not introduced by BL-048) and is a
genuine semantic question: CONNECTED as "all configured links nominal" (current)
versus "all currently-connected links healthy". Because the label is FSM-facing,
the choice needs a deliberate decision; BL-095 tracks it.

### M-4 (clarified this pass) -- inverted-looking stickiness constant names

`_STAY_CONNECTED_BASE` (0.97) and `_STAY_DISCONNECTED_BASE` (0.93) in
`estimators/comms.py` are the shared max / min stickiness bounds, not per-state
baselines, which the names imply. The math is correct. Clarified with a comment
this pass rather than renaming the referenced identifiers.

### L-1 (reviewed) -- degenerate `snr_full_db == snr_floor_db` is a hard step

`capacity_bps` returns full bandwidth at or above the threshold and zero below it
when the ramp endpoints coincide. The sweep called this fail-open; reviewed as an
intentional hard step (a degenerate ramp), and it is pinned by
`test_capacity_degenerate_ramp_is_a_step`. No change.

### L-2 (reviewed) -- `pkg.attempts` increments on a down-link check

The outbox increments `attempts` when a link is simply down, not only on a real
transmission attempt, while the budget-exceeded path does not. A minor diagnostic
inconsistency, not a correctness issue. Noted, not scheduled.

### Cleared (confirmed not bugs)

The sweep explicitly cleared, and this pass confirms: no log-of-nonpositive risk
in `knife_edge_diffraction_db`; the haversine `min(1.0, sqrt(a))` asin guard; the
physically-correct negative `rician_fade_db` (constructive multipath) with its
`log10(0)` guard; no global `numpy.random` leak (every draw goes through the
injected RNG); the `-173.975 dBm/Hz` kTB constant; `log_distance_path_loss_db`
reducing exactly to free space at exponent 2; flush iteration over a snapshot;
and the eviction ordering.

## Findings: core and high-blast surfaces

### HIGH-1 (HIGH, open, BL-090) -- runner returns the raw exception string to the caller

`runner.py:79` builds a caught-worker-error body as `f"[error {cls}: {exc}]"`
(raw `str(exc)`) and only `_truncate`s it before returning it to the MCP caller;
the `args` are redacted but the output body is not. A standard-library exception
can carry a connection string or path (`OSError`, an HTTP client's DSN), so a DB
error during a T0 read (`state_get` / `state_history`) could surface a credential
to any caller, the leak the `server.py` init path already guards against by
reducing a DB error to its class name. `runner.py` is a high-blast surface, so
the remediation (class-only, or a redacted/truncated body) is ADR-gated; BL-090
tracks it. This is the most material finding of the pass.

### HIGH-2 (reviewed + documented this pass) -- uncaught tick exception crashes the server

`tick.py` calls `engine.tick()` without a handler, so a non-finite estimate that
trips `Engine._assert_post_tick_finite` crashes the server process. Validated as
the intended fail-loud design with a real recovery path: `deploy/systemd/nous.service`
sets `Restart=on-failure` / `RestartSec=5s`, so the engine re-initialises within
seconds. The gap was that the contract was undocumented; documented this pass in
the `tick_loop` docstring. No code change.

### MED-1 (MED, open, BL-092) -- `inference_cloud` builds a fresh client per call

`tools/inference.py:94` constructs `AnthropicClient(cfg)` inside `_work()` on
every call rather than reusing the `@lru_cache build_client()` singleton, so each
call allocates a new `httpx` connection pool (a full TLS handshake) and the
per-instance `last_cache_read_input_tokens` cache metric is written on the
thrown-away client and lost, leaving the prompt-cache discipline unobservable.
The cap counter is file-backed so the SC-5 anti-retry-storm property is intact.
The fix (call `build_client`) does not change the observable contract; BL-092.

### MED-2 (MED, open, BL-093) -- `tick_advance` reports a tick count that can undercount

`tools/scenarios.py` `tick_advance` yields to the event loop every 50 ticks, and
the concurrent `tick_loop` can fire its own `engine.tick()` during the yield, so
the engine's `state.tick` / `ts_s` advance by more than the returned
`ticks_advanced`. The docstring acknowledges the extra ticks but the field name
states an exact count. The fix is a field rename (`ticks_requested`) or stopping
the loop during a manual advance; BL-093.

### MED-3 (MED, open, BL-094) -- `Estimator` Protocol omits `rejected_updates`

`estimators/base.py` declares only `name` / `predict` / `update` / `state`, but
three concrete estimators expose `rejected_updates` and `tools/subsystems.py`
reads it directly, so a future Protocol-satisfying replacement would `AttributeError`
in a T0 status tool. `estimators/base.py` is a high-blast surface, so extending
the Protocol is ADR-gated; BL-094 (paired with LOW-4).

### LOW-1 (known open) -- O(N) chain-head recovery scan

`audit.py` `_recover_chain_head` scans the whole audit file on every sink open. A
known-open performance item, already noted in `audit-2026-06-14.md` "out of
scope" (the O(N) audit-head recovery scan); not newly introduced. Tracked there;
a seek-from-end fix is the eventual remediation.

### LOW-2 (LOW, open, BL-096) -- `inference_local` classified T1 despite monotonic counters

`policy.py` places `inference_local` in `_REVERSIBLE_TOOLS` (T1), but it
increments monotonic `local_calls` / `total_tokens` / `total_energy_j` counters
that no compensating call undoes, so the "trivially undone" T1 criterion is
strained, and under `guarded` mode a T1 tool is admitted without an allowlist
match. Reclassifying to T2 would change the guarded-mode contract, so it is
ADR-gated; BL-096 tracks the disposition.

### LOW-3 (LOW, open, BL-092) -- a cap fsync failure is reported as exhaustion

`anthropic_client.py` `CallCap.increment` raises `CapExhausted` on an `os.fsync`
`OSError`, which `InferenceFallback` surfaces as a cap hit, so a transient disk
flush error reads to an operator as genuine exhaustion (`reason: cap_exhausted`).
The fail-closed behaviour is correct (degrade to the local mock); the misleading
`reason` hinders diagnosis. A distinct error type is the fix; high-blast
(`anthropic_client.py`), so paired with BL-092 and ADR-gated.

### LOW-4 (LOW, open, BL-094) -- `decode` return type admits non-string keys

`interop/base.py` declares `decode -> Mapping[str, Any]`, which permits integer
keys; a future CBOR or msgpack adapter returning such a dict would raise
`TypeError` under the tool's `json.dumps`, surfaced as an exception body. Latent
(no current adapter does this). High-blast (`interop/base.py`), ADR-gated; paired
with BL-094.

### DOC-1 (MED, fixed this pass + BL-097) -- the MkDocs nav had drifted, ADRs stopped at 0017

The documentation sweep checked the ADR index at `docs/adr/README.md`, which
`make schema` regenerates, and found it complete. It did not catch that the
hand-maintained `nav:` block in `mkdocs.yml` listed only ADRs 0000 through 0017,
missing every ADR from 0018 to 0054 -- including this audit's own 0053 and 0054.
The live `mkdocs build --strict` gate in this pass surfaced it: the build
succeeds (exit 0) but emits an INFO line for each page absent from the nav,
because a page-not-in-nav is INFO, not WARNING, so `--strict` never failed and CI
never caught the gap. The 37 missing entries were generated from each ADR's H1
title and inserted by hand this pass, so the nav now lists 0000 through 0054.

The same INFO list exposed five further substantive reference pages the nav never
surfaced: three conformance posture docs (`conformance/mqtt.md`,
`conformance/biometrics.md`, `conformance/sensors.md`, each larger than the
already-listed `sosa.md` / `mosa.md`), the 204-line `assumptions.md` modelling
assumptions doc, and `security/bandit-suppressions.md`. All five were folded into
the nav this pass. What remains out of the nav is deliberate: the dated audit and
review logs (this document among them, a record set referenced by path, never
browsed) and the per-scenario showcase pages (reached from the in-nav scenario
gallery `README`).

The systemic fix -- generating the ADR nav block the way the index is already
generated, behind a `--check` drift gate with an allowlist for the intentional
dated / gallery exclusions -- is tracked as BL-097, so a future ADR cannot ship an
out-of-date nav. This is the one finding the sweep missed and validation caught,
which is the argument for keeping the live-build gate in the audit battery rather
than trusting the agent's static read.

## Recommended remediation order

1. BL-090: redact the runner's worker-exception body (HIGH-1, the credential-leak
   surface). High-blast, ADR.
2. BL-091: the comms propagation-link correctness fixes (H-1 zero-capacity `tx`,
   H-2 estimate refresh, M-1 estimate bandwidth, M-2 likelihood boundary).
   Low-blast.
3. BL-092: reuse the `AnthropicClient` singleton and distinguish a cap fsync
   failure from exhaustion (MED-1, LOW-3). High-blast (`anthropic_client.py`),
   ADR.
4. BL-093: make `tick_advance` report an accurate tick count (MED-2).
5. BL-094: extend the `Estimator` Protocol with `rejected_updates` and tighten the
   interop `decode` key contract (MED-3, LOW-4). High-blast Protocols, ADR.
6. BL-095: decide the `comms_state` CONNECTED semantics for multi-link profiles
   (M-3). FSM-facing.
7. BL-096: review the `inference_local` T1 tier classification (LOW-2). ADR if
   reclassified.
