# Audit 2026-06-14: adversarial full-pass plus comms store-and-forward remediation

- **Branch audited:** `claude/eloquent-mccarthy-6xqsxw`
- **Baseline revision:** `2354ce3` (main at the start of the pass)
- **Scope:** an adversarial in-depth read of the whole tree, with a deliberate
  focus on the communication paths and on what happens to outbound traffic when
  comms are degraded or denied. Findings are recorded here; the headline comms
  gap is remediated in the same engagement (BL-077, ADR 0047).
- **Method:** three read-only reconnaissance sweeps (a comms-path trace, two
  independent adversarial correctness sweeps, and a conventions survey), every
  high or notable finding re-verified by hand against the source before it was
  recorded, and the live twin probed to confirm the headline gap in the field.

## Executive summary

The repository is in good health: the quality gates are green, the recent ADRs
(0040 through 0046) are well-tested, and the two independent adversarial sweeps
converged on the same short list of real issues rather than a long tail of
noise. The one finding that rises to operational significance is in the comms
path, and it is the kind of gap a digital twin of an edge appliance should not
carry: the send seam is fire-and-forget, so a transmission on a degraded,
denied, or unknown link is dropped on the spot with no queue, no retry, and no
triage. That is recorded below as COMMS-1 and remediated in this engagement.

The remaining findings are genuine but lower-severity, and several sit on the
high-blast-radius surfaces (`audit.py`, `anthropic_client.py`, `runner.py`) that
the contributor guide reserves for their own ADRs. Those are recorded and left
for a dedicated change rather than folded into this comms-focused pass, in
keeping with the project's blast-radius discipline. Two cross-validated
"findings" turned out to be deliberate design and are dispositioned as rejected,
so the record distinguishes a real gap from a working-as-intended surface.

## Severity legend

- **HIGH** -- operational or integrity impact; fix soon.
- **MEDIUM** -- correctness or legibility gap with a real but bounded blast.
- **LOW** -- cosmetic, performance, or doc drift; fix opportunistically.
- **REJECTED** -- raised by a sweep but verified to be working as designed.

## Findings

### COMMS-1 (HIGH, remediated) -- outbound traffic dropped on a degraded or denied link

`CommsSubsystem.tx` (`subsystems/comms.py`) accepts bytes only on a live link;
on an aged-out, forced-down, or unknown link it returns zero. `comms_send`,
`comms_publish`, and `self_model_publish` all funnel through that seam (the last
two through `tools/publish.py:encode_and_tx`), so a publish during a comms outage
encodes a full standards-shaped message and then discards it, reporting only
`bytes_accepted: 0`. There is no store-and-forward queue, no retry on recovery,
no prioritisation, and no record of the dropped traffic anywhere in the path. A
controller operating autonomously across an intermittent link loses exactly the
situation reports it most needs to deliver.

Validated on the live twin: with all three links denied (aged out), a
`comms_publish` of a CoT event encoded a complete 352-byte message and returned
`bytes_accepted: 0`. The bytes were gone.

**Disposition: remediated.** BL-077 / ADR 0047 add a bounded,
precedence-ordered store-and-forward outbox (`state/comms_outbox.py`) the engine
drains each tick as links recover, plus the `comms_enqueue` / `comms_outbox` /
`comms_flush` tools. The full delay-tolerant-networking layer (multi-hop
custody, BPv7, mesh, replay) remains scoped to BL-056.

### AUD-1 (MEDIUM, open) -- audit chain head advances before the fsync delta is checked

In `AuditLogger.write` (`audit.py`), the in-memory `_chain_head` is advanced
immediately after `self._log.info(...)` emits the record, and only then does
`_sync_fsync_failure_state()` poll the handler for a silent fsync failure. The
emit writes the line into the file, so the on-disk chain stays internally
consistent and `audit_verify` still walks cleanly; the real exposure is narrow:
on a silent fsync failure followed by a crash, the in-memory head references a
record that reached the page cache but may not be durable. The `degraded` flag
is set in that window, which is the operator's signal, so this is a
durability-versus-head-advance subtlety rather than the silent chain corruption
one sweep framed it as. `audit.py` is a high-blast surface; left for a dedicated
ADR that moves the head advance inside the clean-fsync branch.

### CAP-1 (MEDIUM, open) -- cap status reads fail-open while the increment fails-closed

`CallCap.increment` (`anthropic_client.py`) raises `CapExhausted` on a corrupt
counter file (fail-closed), but `CallCap.peek` returns `(0, cap)` on the same
corruption (fail-open), so `anthropic_cap_status` can report the cap as fully
available immediately before an `inference_cloud` call trips on the corruption.
The fallback ladder still degrades to the local mock, so there is no cap bypass
and a controller always gets an answer; the gap is that the T0 status tool
misreports. High-blast surface; recorded for the cloud-path ADR that already
owns this file.

### DB-1 (MEDIUM, remediated) -- silent database init failure

`Nous.__init__` (`server.py`) swallowed any `init_db` exception and fell back to
a memory-only `StateTransitionLog(None)` with no log line, no degraded marker,
and no surfacing in `device_info` or `device_health`. An operator who
misconfigures `NOUS_DB_URL` got no signal that the transition history was not
being persisted until they noticed `state_history` was memory-backed.

**Disposition: remediated.** BL-078 adds an `init_error` to
`StateTransitionLog` that separates the intentional memory-only mode from a
failed init, a `degraded` property and a `status()` read, a stderr echo on the
failure, and a `persistence` block in `device_info`.

### RLD-1 (MEDIUM, remediated) -- profile reload did not reset failsafe streaks or capability claims

`Engine.reload_profile` rebuilt every subsystem and estimator but left the
`FailsafeArbiter` debounce streaks and `state.last_capabilities` intact. A
streak part-way to an auto-safe carried across a reload onto a profile that may
define a different threshold, and a tool reading `snapshot` between the reload
and the next tick saw capability claims computed from the old profile's physics.

**Disposition: remediated.** BL-079 rebuilds the arbiter from
`_FAILSAFE_CONDITIONS` (fresh streaks) and recomputes the capabilities from the
rebuilt estimators inside `reload_profile`, so both restart against the new
physics. The `SafetyEnforcer` violation counters stay cumulative by design (a
lifetime diagnostic, not reload-scoped).

### RUN-1 (MEDIUM, open) -- runner exception path leaves `exit_code` unset

In `runner.py`, the policy-denied path stamps `exit_code=1`, but the caught
exception path writes the audit record with `exit_code` defaulting to `None`, so
an audit consumer cannot cleanly distinguish "tool returned an error body" from
"tool raised and the runner caught it." `runner.py` is a high-blast surface;
recorded for its own ADR.

### COMMS-2 (LOW, open) -- silent link age-out emits no event

When a link crosses `max_age_s` in `CommsSubsystem.step`, `connected` flips to
false and throughput drops to zero with no audit record or tick event at the
moment of age-out. The next `derive_state` reports the new comms state, so the
condition is observable, but the transition itself is not stamped.

### COMMS-3 (LOW, open) -- `tx` records throughput as bytes times eight, not a rate

`Link.tx` sets `throughput_bps = n_bytes * 8` with no division by a time delta,
so the field the comms particle filter consumes is a per-transmission bit count
labelled as a rate. The docstring already calls it a coarse figure; tightening
it to a real rate would change the estimator's input signal and is left out of
this additive pass to avoid estimator-test churn.

### COMMS-4 (LOW, open) -- comms estimator log-throughput z-score collapses to a constant

In `estimators/comms.py`, the log-throughput z-score divides the log residual by
`sigma / max(expected, 1.0)`, which simplifies to the constant
`_THROUGHPUT_OBS_SIGMA_FRAC` whenever the expected throughput exceeds one. The
intermediate `sigma` is effectively dead and the likelihood's spread does not
scale with signal strength. Likely benign for the binary connected/disconnected
estimate, but the math reads as an accident rather than a choice.

### FRESH-1 (LOW, open) -- a fresh engine cannot encode interop until it ticks

`interop/base.py:resolve_ts` accepts `ts_s = 0.0` verbatim, so before the engine
has ticked, any adapter encode sees an age of roughly the Unix epoch and raises
`StaleEstimateError`. The self-model publish path sidesteps this by stamping the
live wall clock, and a running engine has a non-zero sim clock, so the practical
exposure is a freshly built engine in a test or an `at_min: 0` scenario step.

### ASSESS-1 (LOW, open) -- Monte Carlo endurance discards the sample median

In `self_model/assess.py`, the Monte Carlo branch computes the 5th, 50th, and
95th percentiles from the sample but returns `Capability.p50 = point_min` (the
deterministic estimate) and discards the sample median. The published band
therefore mixes sampled tails with a deterministic centre. A semantic
inconsistency in the calibration story rather than a crash.

### TICK-1 (LOW, open) -- `tick_overruns` written into the capability dict

`tick_loop` writes `last_capabilities["tick_overruns"]` once on loop teardown,
mixing a loop metric into the dict that `self_model_assess` and `device_health`
read as capability claims. It is a teardown artefact, not a per-tick pollution,
so the impact is small.

### ITP-1 (LOW, open) -- misleading stale-estimate message on an invalid `max_age`

`assert_fresh` raises `StaleEstimateError(adapter, 0.0, max_age_s)` when
`max_age_s <= 0` or is NaN, so the message reports the estimate as brand new
rather than naming the misconfiguration. Cosmetic.

### DOC-1 (LOW, fixed) -- inference fallback docstring omitted the DEGRADED route

The `inference_fallback.py` module docstring listed CONNECTED and LIMITED in the
ladder but never named DEGRADED, although the code routes DEGRADED straight to
the local mock. Fixed in this engagement.

### DOC-2 (LOW, fixed) -- stale tool and test counts in STATUS.md

STATUS.md carried a thirty-seven and a forty-three tool figure and an 812-test
figure. Reconciled to the current forty-six tools and 903 tests in this
engagement.

### DOC-3 (LOW, open) -- backlog drift on the interop adapters

`docs/backlog.md` BL-036 says the MQTT adapter is "not yet wired" though the
encode/decode surface ships and is registered (only broker connectivity is
absent), and BL-033 carries `pynmea2` as a dependency the hand-rolled NMEA
adapter never imports. Both are backlog-text drift, not code defects.

### REJECTED -- verified working as designed

- **`inference_request` in `_STATEFUL_TOOLS` is not an orphan.** ADR 0033 (and
  the ADR 0034 note) deliberately forward-classify it as the redundant twin of
  `inference_cloud` under the additive-surface rule; `classify` also defaults
  any unknown name to STATEFUL, so the entry is intentional, not stale.
- **Cap incremented before the API call is intentional.** The SC-5 contract
  charges the slot first so a retry storm cannot bypass the daily cap; the
  comment records this.
- **`is_live` (`<=`) versus `step` (`>`) at `max_age_s` is a benign one-tick
  window**, not an off-by-one defect: a link exactly at `max_age_s` reads live
  for the one tick before `step` pushes it past the bound.

## Recommended remediation order

1. DB-1: surface the database-init degradation in `device_info` (done, BL-078).
2. RLD-1: reset the failsafe streaks and capability cache on reload (done, BL-079).
3. AUD-1, RUN-1, CAP-1: the high-blast surfaces, each behind its own ADR.
4. The LOW findings: opportunistic, batched into the next cadence pass.

## Out of scope

Performance work (the BL-073 Monte Carlo tick cost, the O(N) audit-head recovery
scan), the full DTN layer (BL-056), and the propagation-aware comms model
(BL-048) are noted but not addressed here.
