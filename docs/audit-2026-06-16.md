# Audit 2026-06-16: comms diagnostics, persistence, and self-model legibility

A focused adversarial pass at HEAD (`658fbed`, after PR #167), over the surfaces
that changed across PRs #160 through #167: the per-cause traffic diagnostics
(BL-108), the EMCON rejected-default legibility and MCP-path coverage (BL-104 /
BL-105), the DtnStore load/save split (BL-101), the outbox O(1) dedup (BL-102),
and the shared publish path. Each finding below was validated by hand against the
code before it was recorded; the goal was to separate genuine gaps from
over-statements, the same discipline the prior passes used.

The headline result is a legibility gap the BL-108 work surfaced but did not
close: the comms layer now records why a transmission was refused
(`last_tx_reason`), but the fire-and-forget send and publish tools still drop a
rejected transmission without reporting that reason, so a controller cannot tell
a forced-down link from a below-SNR (zero-capacity) link on a dropped send. A
second, smaller miss is that the two read tools whose counters gained the new
per-cause maps do not yet name those maps in their tool-contract docstrings. The
docstring gap is fixed in this same change; the send/publish reason gap and a
minor self-model status inconsistency are recorded as backlog items for their own
focused changes, consistent with how the 2026-06-15 structural findings were
handled.

The larger result is reassurance: the highest-risk recently-shipped code held up.
The outbox O(1) dedup keeps `_queued_ids` in exact sync with the package list on
every add and remove path, the per-cause defer and drop maps attribute correctly
(counted once per link per flush, budget-holds excluded, runtime-only as ADR 0070
states), the new maps genuinely reach the controller through the read tools, and
the DtnStore load/save split records a read fault distinctly from a write fault
without leaking an exception message through a T0 read.

## Confirmed clean

The outbox dedup (BL-102) is sound: `_remove` is the single removal funnel and
discards the bundle id from `_queued_ids` while adjusting `_queued_bytes`, and
every remove path (eviction, expiry purge, flush expiry, flush delivery) routes
through it, so the O(1) `_is_duplicate` cannot leak a stale id or reject a valid
re-enqueue.

The per-cause diagnostics (BL-108) attribute correctly. The outbox closes a link
on its first deferral, so each cause is counted once per link per flush and a
later knock-on package for the same link is not re-counted; a budget-held package
bumps no attempt and no cause, exactly as ADR 0070 states. The mesh `_count_drop`
is reached from all four drop sites (`max_hops`, `forward_loss`,
`retry_exhausted`, `store_overflow`) and from none other, and neither
`defer_causes` nor `drop_causes` is persisted, so the within-process attribution
matches the ADR. Both maps reach the controller: the `comms_outbox` and
`dtn_mesh` read tools return `status()` directly, and a live `status()` carries
the maps in its counters.

The DtnStore split (BL-101) is correct: `degraded` is true on either a load or a
save failure, `status()` surfaces both pairs, and only the exception class is
recorded, so a corrupt restore reads as a read fault with the save counters
untouched and no message leaks through the `dtn_mesh` read. The engine tick
integration is coherent: the outbox drain, the mesh step, and the dirty-gated
DtnStore save run in order, and the runtime-only maps never enter the snapshot.
The publish shaping (`_publish_shape`) carries no raw coordinates outside the CoT
path, so a minimising posture has nothing to leak through the MQTT, SensorThings,
or STANAG envelopes. EMCON default handling (BL-104) reports a rejected default
distinctly from a chosen `unrestricted` and from an unconfigured default.

No finding reopens a 2026-06-14b, 2026-06-15, or 2026-06-15b item.
